# ARTEMIS — POLICY OF DOCUMENTATION STORAGE AND PLACEMENT

Статус: reference / historical governance-support material  
Дата: 2026-04-19  
Версия: v1.0  
Назначение: зафиксировать historical/supporting policy по размещению документации между GitHub, ChatGPT и Google Drive. Текущим canonical source of truth для documentation governance является `docs/DOCUMENTATION_SYSTEM.md`.

---

## 1. ЦЕЛЬ ДОКУМЕНТА

ARTEMIS требует не только иерархии документации по типам (`canonical / working / audits / archive`), но и чёткого распределения документации по носителям.

Без этого возникают три критические проблемы:
- competing source of truth;
- drift между кодом, документами и рабочими инструкциями;
- потеря управляемости при активной фазе разработки.

Цель этой политики — зафиксировать единую модель размещения документов, чтобы:
- каждый документ имел понятное постоянное место;
- было ясно, какой носитель имеет приоритет при конфликте;
- рабочие материалы не подменяли canonical layer;
- совместная редактура не разрушала системную целостность проекта.

---

## 2. БАЗОВЫЙ ПРИНЦИП

В ARTEMIS не должно существовать нескольких равноправных источников истины для одной и той же сущности.

Распределение ролей носителей:
- **GitHub** — официальный versioned source of truth для системной документации проекта;
- **ChatGPT** — operational orchestration layer для мышления, синтеза, агентных инструкций и подготовки решений;
- **Google Drive** — collaborative workspace для ручной совместной подготовки, редактуры и промежуточных материалов.

Если один и тот же документ одновременно существует в нескольких местах, обязательно должен быть явно определён:
- основной носитель;
- вторичный носитель;
- момент синхронизации;
- правило разрешения конфликтов.

---

## 3. ПРИОРИТЕТ НОСИТЕЛЕЙ

### 3.1 Общий приоритет

При конфликте между носителями действует следующий приоритет:

1. **GitHub canonical docs**
2. **GitHub working docs**
3. **ChatGPT operational documents**
4. **Google Drive working documents**
5. **archive / snapshots / historical copies**

### 3.2 Что это означает practically

- Если в ChatGPT сформулировано решение, но оно не перенесено в canonical docs GitHub, это решение не считается окончательно принятым.
- Если в Google Drive ведётся рабочая редактура, но финальный результат не зафиксирован в GitHub там, где это требуется, Drive-версия не считается source of truth.
- Если старый snapshot-документ противоречит текущему canonical doc, приоритет всегда у текущего canonical doc.

---

## 4. ЧТО ХРАНИТЬ В GITHUB

GitHub является обязательным местом хранения для всех документов, которые:
- определяют архитектуру проекта;
- фиксируют правила системы;
- влияют на runtime, data contract, release semantics или product scope;
- должны быть reviewable через commit history;
- должны синхронизироваться с кодом, тестами, CI и release discipline.

### 4.1 В GitHub обязательно хранить

#### Canonical layer
- `README.md`
- `docs/ARTEMIS_CONCEPT.md`
- `docs/ARTEMIS_MASTER_PROMPT.md`
- `docs/ARTEMIS_PRODUCT_SCOPE.md`
- `docs/PROJECT_STRUCTURE.md`
- `docs/PROJECT_PHASES.md`
- `docs/PRIORITIES.md`
- `docs/DATA_CONTRACT.md`
- `docs/CONTROLLED_RELEASE_DECISION.md`

#### System working docs
- UI/UX system documents
- UI/UX component maps
- AI strategy, если она влияет на продуктовые и системные решения
- migration/cleanup plans по repo и docs system
- release/readiness/stabilization documents
- backend/frontend contract notes, если они используются в процессе разработки и должны быть связаны с кодом

#### Audit layer
- audits состояния repo, runtime, CI, data contract, docs sync
- quality checks, architecture audits, release audits

#### Archive layer
- snapshot-версии документов;
- historical reference-файлы, которые больше не являются operational source of truth, но нужны для traceability.

### 4.2 Что нельзя оставлять только вне GitHub

Нельзя хранить только в ChatGPT или только в Google Drive документы, которые определяют:
- architecture boundaries;
- runtime entrypoints;
- data contract;
- ETL/publish semantics;
- release rules;
- product boundaries;
- phase order;
- documentation hierarchy;
- допустимую роль AI в системе ARTEMIS.

Если такой материал родился вне GitHub, он должен быть перенесён в GitHub до того, как начнёт считаться действующим правилом проекта.

---

## 5. ЧТО ХРАНИТЬ В CHATGPT

ChatGPT не является официальным системным хранилищем проекта. Его роль — operational control room.

### 5.1 В ChatGPT допустимо и целесообразно хранить

#### Агентные и системные инструкции
- operating instructions для Codex;
- role prompts для ChatGPT, Claude и других агентов;
- audit prompts;
- decomposition rules;
- рабочие шаблоны задач и форматы запросов.

#### Операционные synthesis-документы
- сводки текущего состояния проекта;
- conflict maps между repo и docs;
- next action packs;
- planning notes;
- предварительные governance-решения;
- временные архитектурные или продуктовые synthesis before commit.

#### Временные рабочие конструкции
- draft-версии будущих документов;
- варианты структуры docs;
- подготовленные rename/move packages;
- аналитические сборки для последующего переноса в GitHub.

### 5.2 Ограничения для ChatGPT

ChatGPT не должен быть единственным местом хранения для:
- системных правил проекта;
- окончательных архитектурных решений;
- действующего data contract;
- release-governance;
- phase governance;
- окончательно утверждённого product scope.

Если документ из operational слоя начинает играть системообразующую роль, он должен быть переведён в GitHub.

---

## 6. ЧТО ХРАНИТЬ В GOOGLE DRIVE

Google Drive — это collaborative workspace, а не главный системный реестр проекта.

Он нужен для документов, где важнее:
- ручная совместная редактура;
- длинный текстовый workflow;
- комментарии и согласования;
- таблицы и списки;
- редакторская и исследовательская подготовка.

### 6.1 В Google Drive рекомендуется хранить

#### Google Docs
- исторические исследования;
- редакторские документы;
- long-form narrative drafts;
- тексты stories и courses до финализации;
- материалы для ручной вычитки и обсуждения.

#### Google Sheets
- контентные реестры;
- source logs;
- tracking источников;
- списки объектов;
- editorial pipelines;
- moderation checklists;
- таблицы ручной подготовки данных.

#### Google Slides
- презентации;
- концептуальные материалы;
- visual/narrative decks;
- внешние коммуникативные материалы.

### 6.2 Ограничения для Google Drive

Drive не должен быть единственным местом хранения для:
- архитектурных норм проекта;
- release policy;
- canonical product rules;
- runtime/data/public contract;
- documentation hierarchy;
- решений, напрямую определяющих кодовую систему.

Drive может использоваться как предфинальный этап подготовки, но не как финальный authority-layer для системных документов.

## 7. ПРАВИЛА ПЕРЕНОСА ДОКУМЕНТОВ МЕЖДУ НОСИТЕЛЯМИ

### 7.1 ChatGPT → GitHub

Перенос обязателен, если документ из ChatGPT:
- определяет правило проекта;
- фиксирует архитектурное решение;
- меняет scope;
- меняет phase order;
- вводит новый data/release/runtime contract;
- становится опорой для patch, CI или repo-изменений.

Минимальный порядок:
1. подготовить структурированный документ или diff-пакет;
2. определить целевой файл в GitHub;
3. проверить, canonical это слой или working;
4. внести документ в repo;
5. при необходимости обновить связанные canonical docs.

### 7.2 Google Drive → GitHub

Перенос обязателен, если результат работы в Drive:
- перестал быть просто черновиком;
- утверждён для operational use;
- влияет на разработку, релиз или архитектурные решения;
- нужен как постоянный reference для команды и агентов.

Типовой порядок:
1. завершить редактирование и согласование в Drive;
2. извлечь стабильную финальную версию;
3. определить, это working doc или canonical doc;
4. перенести в GitHub;
5. оставить в Drive либо editable original, либо ссылочную копию без статуса source of truth.

### 7.3 GitHub → Google Drive

Допустимо, если:
- нужен удобный human-edit workflow;
- требуется совместная редактура текста перед следующей версией;
- создаётся презентационная или редакторская производная.

Но при этом GitHub-версия должна сохранять статус основной, если речь идёт о системном документе.

### 7.4 GitHub → ChatGPT

Допустимо только как operational extraction:
- для synthesis;
- для анализа;
- для построения next action;
- для подготовки новых решений.

Такой перенос не меняет статус основного носителя. Оригиналом всё равно остаётся GitHub.

---

## 8. ПРАВИЛО ЕДИНОЙ ФУНКЦИИ ДОКУМЕНТА

Один документ должен иметь одну главную функцию и один основной носитель.

Недопустимо:
- держать один и тот же активный документ одновременно как равноправный official doc в GitHub и Drive;
- вести архитектурную политику в ChatGPT без фиксации в repo;
- использовать Drive-файл как реальный canonical doc, если GitHub-файл уже существует;
- хранить несколько параллельных active-версий одного документа без явного статуса.

---

## 9. РЕКОМЕНДУЕМАЯ МОДЕЛЬ ДЛЯ ARTEMIS

### 9.1 GitHub = official system of record

Там должны находиться:
- все canonical docs;
- repo-related working docs;
- data/release/runtime policies;
- docs structure and governance rules;
- audits, связанные с кодом и системой;
- archive snapshots.

### 9.2 ChatGPT = operational control room

Там должны находиться:
- мастер-промпты и роли агентов;
- операционные регламенты работы с ИИ;
- synthesis-документы;
- конфликт-карты;
- промежуточные planning packs;
- временные документы до их закрепления в repo.

### 9.3 Google Drive = collaborative content workspace

Там должны находиться:
- контентные research-документы;
- редакторские наборы;
- рабочие таблицы;
- source tracking;
- совместно редактируемые narrative/concept materials;
- презентационные материалы.

---

## 10. КТО И КОГДА ДОЛЖЕН ПЕРЕНОСИТЬ ДОКУМЕНТЫ

### 10.1 Инициатор переноса

Инициатором переноса является тот, кто создаёт или существенно меняет документ и видит, что его функция изменилась.

Это означает:
- если ChatGPT сформировал системное решение, оно должно быть подготовлено к переносу в GitHub;
- если в Drive согласован документ, который теперь управляет проектной практикой, он должен быть перенесён в GitHub;
- если GitHub-документ временно выводится в Drive для совместной редакции, после согласования его новая версия должна вернуться в GitHub.

### 10.2 Момент обязательного переноса

Перенос обязателен, когда документ:
- перестаёт быть чисто черновым;
- начинает влиять на рабочие решения;
- используется для постановки задач;
- используется для оценки готовности;
- используется как аргумент в архитектурных или продуктовых спорах;
- определяет ограничения для кода, данных или интерфейса.

---

## 11. РАЗРЕШЕНИЕ КОНФЛИКТОВ

Если версии документа или смысла расходятся, применяется такой порядок:

1. определить, какой носитель должен быть основным для этой сущности;
2. если есть GitHub canonical version — она главная;
3. если есть только working materials, нужно определить, где должен жить финальный authority-layer;
4. устаревшие версии переводятся в archive или удаляются из active layer;
5. после разрешения конфликта обновляется только один основной active документ.

При конфликте запрещено оставлять несколько активных competing versions без пометки и без решения.

---

## 12. МИНИМАЛЬНЫЙ GOVERNANCE-CHECK ДЛЯ ЛЮБОГО ДОКУМЕНТА

Перед созданием или переносом документа нужно ответить на 5 вопросов:

1. Какова единственная главная функция документа?
2. Какой носитель для него основной?
3. Является ли он canonical, working, audit или archive?
4. Может ли он влиять на код, runtime, release или product scope?
5. Нужна ли синхронизация с GitHub после его подготовки?

Если на вопросы 2, 4 или 5 нет ясного ответа, документ ещё не готов к operational use.

---

## 13. ЧТО НУЖНО СДЕЛАТЬ В ARTEMIS СРАЗУ ПОСЛЕ ПРИНЯТИЯ ЭТОЙ ПОЛИТИКИ

1. Зафиксировать эту политику в GitHub как системный документ docs-governance.
2. Уточнить в `PROJECT_STRUCTURE.md`, что это основной документ по размещению документации по носителям.
3. Провести cleanup текущего активного doc-layer:
   - вынести transitional/snapshot docs из active root;
   - отделить working docs от canonical layer;
   - явно отметить archive.
4. Зафиксировать, какие агентные инструкции остаются только в ChatGPT, а какие должны дублироваться в GitHub.
5. Определить набор рабочих документов, для которых Google Drive является допустимым collaborative workspace.

---

## 14. ФИНАЛЬНОЕ ПРАВИЛО

GitHub хранит правила системы.  
ChatGPT хранит оперативное мышление и orchestration.  
Google Drive хранит совместную подготовку материалов.  

Ни один из этих слоёв не должен подменять другой.

Если документ начинает определять ARTEMIS как систему, его место — в GitHub.
